# Variant windows map: A03_26 vs A04_28

Дата: 2026-04-24 (UTC).  
Метод: статическое сравнение Intel HEX в диапазоне `0x4000..0xBFFF`.

## 1) Что подтверждено

- Оба образа имеют общий платформенный входной каркас в верхней странице: векторы `LJMP` на `0x4000/0x4006/0x400C/0x4012/0x4018/0x401E` совпадают (`4100, 4176, 41D0, 492E, 4954, 497A`) — **подтверждено**.
- Для окна `0x4000..0x41FF` совпадает `508/512` байт (~99.2%) — **подтверждено**.
- Большое общее окно `0x420A..0x4FFF` практически без различий — **подтверждено**.
- Отличия A03/A04 нужно трактовать как набор вариантных окон и таблиц, а не как полностью разные платформы — **вероятно (сильная гипотеза)**.

## 2) Сводка по сходству

- Полный диапазон `0x4000..0xBFFF`: совпадает `8366/32768` байт (25.53%).
- Ранний platform-frame (`0x4000..0x41FF`): совпадает `508/512` байт.
- Явные общие (длинные) окна:
  - `0x4000..0x41F0`
  - `0x420A..0x4FFF`
  - `0xB49C..0xBFFD`

## 3) Вариантные окна (практически важные)

Ниже — диапазоны, где различия стабильны и воспроизводимы при сравнении.

### 3.1 Окна вокруг `0x5000+` (интерес для совместимой прошивки)

- `0x5000..0x50FF`: смешанное окно (178 совп./78 различ.) — **подтверждено**.
- `0x5100..0x51FF`: смешанное окно (175/81) — **подтверждено**.
- `0x5200..0x52FF`: уже выраженно вариантное (44/212) — **подтверждено**.

Интерпретация: около `0x5000+` расположен runtime-пакетный/буферный слой + связанные таблицы параметров, где A03/A04 расходятся именно как вариантная конфигурация и часть обслуживающей логики — **вероятно**.

### 3.2 Плотные вариантные зоны

- Крупный почти сплошной вариантный пояс: `0x5400..0xB49B` (много длинных diff-сегментов).
- Это не опровергает общий platform-core: каркас старта и часть инфраструктуры совпадают.

## 4) Табличные/калибровочные параметры

Что можно сказать на текущем шаге:

- Наличие плотных отличий в окнах `0x5000+` и далее указывает на массивы параметров/таблиц (уставки/карты/строки/индексы) — **гипотеза**.
- Для строгого разделения «код vs таблицы» требуется следующий шаг: линейный дизассемблер + классификация участков как исполняемых/данных — **подтверждено как ограничение**.

## 5) Ограничения

- Текущий анализ — статический байтовый diff без полноценного CFG.
- Точная семантика каждого вариантного окна не доказана.
- Утверждения о «калибровках» пока только рабочая гипотеза.

## 6) Практическое применение для новой прошивки

Рекомендованный подход:

1. Сохранять единое platform-core (инициализация, очередь, snapshots, builders).
2. Вынести вариантность в конфиг-окна/таблицы (в первую очередь вокруг `0x5000+`).
3. Держать отдельный map-файл вариантных окон для автоматической сборки профилей A03/A04.
