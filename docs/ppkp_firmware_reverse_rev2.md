# Техническая база по реверс-инжинирингу ППКП-01Ф (редакция 2)

**Дата:** 23.04.2026  
**Назначение:** опорный документ для проектирования новой прошивки, продолжения реверса и выделения платформенного ядра.

---

## 1) Краткое резюме (executive summary)

- Исследованные `.PZU` — это **верхняя прикладная страница** платформы в диапазоне `0x4000–0xBFFF`, а не полный ROM.
- Нижний системный слой (ниже `0x4000`) отсутствует в доступных образах, но явно используется через вызовы в низкие адреса.
- Архитектура подтверждена как **событийно-ориентированная**: изменения фиксируются, ставятся в очередь и отправляются пакетами.
- Основа логики — **объектная модель** с таблицей записей фиксированного размера (16 байт), индексами, битовыми картами активности и допустимости.
- По расширенному набору образов подтверждено сильное разделение на:
  - **Platform Core** (стабильное ядро),
  - **Variant Configuration** (узкие окна вариантных данных под объект/здание).

## 4) Формат `.PZU` и контроль целостности

- Формат: **Intel HEX**.
- Объем полезных данных: **32768 байт**.
- Диапазон: `0x4000–0xBFFF`.
- Завершающая запись: `:00000001FF`.
- Практически значимый факт: в конце образа присутствует 16-битная аддитивная сумма (mod 65536), перед ней — байт `0x01`.

**Следствие:** при любой модификации бинарных данных контрольную сумму нужно пересчитывать.

## 5) Подтвержденная архитектурная модель

1. Нижний ROM-слой: системные сервисы/драйверы/часть транспорта.
2. Верхняя страница (`0x4000–0xBFFF`): объекты, UI, очереди, packet builders.
3. Детектор изменений сравнивает новые/старые снимки.
4. Изменения маркируются битами в очереди.
5. Для каждого класса изменений строится свой пакет.
6. Пакет передается в нижний сервис через буфер в области `0x5006+`.

### 5.1 Стартовые векторы (типовой каркас)

```
4000: LJMP 4100h
4006: LJMP 4176h
400C: LJMP 41D0h
4012: LJMP ...
4018: LJMP ...
401E: LJMP ...
```

Совпадение первых трех векторов на большом наборе образов подтверждает платформенный каркас участка `0x4000–0x41DF`.

## 7) Объектная модель

- Размер записи объекта: **16 байт**.
- База: `0x343D`.
- Ключевые индексы:
  - `0x3298` — текущий выбранный объект,
  - `0x3295` — сканирующий индекс при обходе/нормализации.

### 7.1 Поля object record (рабочая интерпретация)

| Смещение | Интерпретация | Статус |
|---|---|---|
| `+0` | тип/класс объекта | вероятно близко к подтверждению |
| `+1` | компактный код состояния | вероятно |
| `+2` | индекс связанного ресурса/группы | вероятно |
| `+3` | режим объекта (младшие 2 бита) | вероятно |
| `+4` | параметр/уровень | гипотеза |
| `+5` | ссылка на связанный объект/группу | гипотеза |
| `+6` | дополнительный статусный байт | гипотеза |
| `+7..+15` | не расшифровано | неизвестно |

## 8) UI/строки/форматирование

Подтвержденные функции:

- `0x6F83` — очистка текстового буфера,
- `0x6FB8` — добавление символа,
- `0x7020` — вывод строки ROM до `0xFF`,
- `0x6E89` — выбор строки по индексу через таблицу `0x6176`.

Наличие BCD/десятичного/hex форматтеров подтверждает диагностический UI с числовыми значениями, а не только со строками.

## 9) Подсистема изменений, очереди и packet runtime

### 9.1 Детектор изменений

Логика уровня модели:

```c
if (block_3110_changed) queue.bit0 = 1;
if (block_3128_changed) queue.bit1 = 1;
if (code_32A2_changed) queue.bit2 = 1;
if (payload_3153_changed) queue.bit3 = 1;
```

### 9.2 Типы builder-функций

- `0x8999`: пакет блока `0x3110`.
- `0x89B1`: пакет блока `0x3128`.
- `0x89C9`: пакет 1-байтного event code (`0x329F`).
- `0x89DD`: пакет 3-байтного payload (`0x3150..3152`).
- `0x8989`: служебный/стартовый пакет (вероятно).

### 9.3 Короткий событийный канал

Связка `0x32A2 + 0x3153..3155` — выделенный служебный канал. Зафиксированы коды `0x02`, `0x83`, `0x84`, `0x85`; сценарий `0x85 + 55 55 55` похож на импульсную команду/маркер с таймерным авто-сбросом.

## 11) Platform Core vs Variant Configuration

Сравнения образов показывают очень малые различия внутри одной ветки (десятки байт, а иногда единицы).

**Практический вывод:**
- ядро платформы стабильно,
- вариантность должна выноситься в конфигурационные таблицы/окна,
- новую прошивку целесообразно делать как платформу + конфигурация, а не как отдельные форки.

## 12) Рекомендуемая архитектура новой прошивки

### Слои

1. **HAL / Transport**: MCU init, таймеры, UART/RS-485, драйверы, загрузчик.
2. **Platform Core**: object runtime, bitmaps, scheduler, queue, snapshots, packet builders.
3. **Variant Configuration**: таблицы объектов/связей/строк/режимов/карт оборудования.
4. **Application Logic**: сценарии конкретного прибора.

### Непременные принципы

- событийный обмен через очередь;
- snapshot + mirror + delta-send;
- объектная модель фиксированного формата;
- явные enum/диапазоны для event codes и object fields;
- документированный конфиг-формат вместо непрозрачных бинарных окон.

## 13) Приоритетный план продолжения реверса

### Этап 1 (высокий приоритет)

1. Расшифровать поля object record `+4..+15`.
2. Построить карту всех обращений к `0x343D + idx*16` по смещениям.
3. Декодировать таблицу `0x7430` и область `0x3165` (группы/связи).

### Этап 2

4. Полностью трассировать жизненный цикл event code (`0x32A2`).
5. Привязать payload-байты к сценариям UI/управления.
6. Сверить поля snapshots `0x3110/0x3128` с сигналами проекта.

### Этап 3

7. Восстановить wire-format протокола через:
   - анализ загрузчика,
   - снятие обмена с реального прибора,
   - (при наличии доступа) дамп нижнего ROM `<0x4000`.

## 16) Что это дает для практического проектирования

1. Проектировать новую прошивку как **совместимый верхний runtime**, а не «перепаковку» бинарника.
2. Закрепить стабильный **Platform Core API** для object runtime, snapshots, queue и packet builders.
3. Вынести объектные таблицы, связи и строки в **явный variant-конфиг** (версионируемый и проверяемый).
4. Подготовить сервис обновления с адресацией устройства и страничной записью, совместимый с текущими практиками обслуживания.

## Заключение

Текущая база уже достаточна для старта новой реализации: архитектурный контур, модель объектов, механизм обмена и платформенное разделение «ядро/вариант» определены.
