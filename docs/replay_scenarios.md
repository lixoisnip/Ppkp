# Replay scenarios для стендовой проверки

Цель: практично проверить гипотезы по detector/queue/builders и вариантным окнам A03/A04/2019.

## S1. Изменение snapshot `0x3110`

- **Цель:** подтвердить постановку флага queue и вызов builder snapshot-1.
- **Области/адреса:** `0x3110`, `0x329C`, `0x329D`, builder `0x8999`, буфер `0x5008+`.
- **Что ожидается:** после изменения байта в `0x3110` появляется pending-бит в `0x329C`, затем формируется пакет в `0x5008+`.
- **Что подтвердит/опровергнет:**
  - Подтверждает detector и привязку `build_3110`.
  - Опровергает, если изменение `0x3110` не приводит к активности queue.

## S2. Изменение snapshot `0x3128`

- **Цель:** отличить второй тип snapshot-пакета от первого.
- **Области/адреса:** `0x3128`, `0x329C`, `0x329D`, `0x89B1`, `0x5008+`.
- **Ожидание:** отдельный бит queue/отдельный тип builder.
- **Критерий:** другой payload/selector по сравнению с S1.

## S3. Смена текущего объекта

- **Цель:** проверить роль `0x3298` как текущего индекса объекта.
- **Области/адреса:** `0x3298`, `0x343D + idx*16`, helper `0x6759/0x67AD/0x6C92`.
- **Ожидание:** смена `idx` приводит к обращению к другому блоку записи объекта.
- **Критерий:** изменяются читаемые значения полей объекта при неизменном коде сценария.

## S4. Генерация event code

- **Цель:** подтвердить канал `0x32A2 -> 0x329F -> builder`.
- **Области/адреса:** `0x32A2`, `0x329F`, `0x329C`, `0x89C9`.
- **Ожидание:** изменение event code создает короткий пакет (1 байт полезных данных).
- **Критерий:** queue-bit поднимается/сбрасывается, в буфере меняется event-поле.

## S5. Генерация payload `3B`

- **Цель:** подтвердить ветку `0x3153..3155` / рабочая копия `0x3150..3152`.
- **Области/адреса:** `0x3153..0x3155`, `0x3150..0x3152`, `0x329C`, builder `0x89DD`.
- **Ожидание:** формируется пакет с 3-байт payload.
- **Критерий:** байты в `0x5008+` меняются согласованно с входными 3 байтами.

## S6. Проверка цикла builders

- **Цель:** доказать round-robin/селекторную логику в `0x329D`.
- **Области/адреса:** `0x329C`, `0x329D`, `0x88C4..0x8975`.
- **Ожидание:** при нескольких одновременно активных битах queue builders вызываются последовательно.
- **Критерий:** последовательные разные типы пакетов без потери pending-флагов.

## S7. A03 vs A04: вариантные окна `0x5000+`

- **Цель:** проверить, что различия — это вариантные окна/таблицы, а не другой протокол.
- **Области/адреса:** `0x5000..0x52FF` и общий каркас `0x4000..0x41FF`.
- **Ожидание:** последовательность событий одинакова, но конкретные поля/константы пакетов различаются.
- **Критерий:** одинаковая high-level реакция при различном низкоуровневом payload.

## S8. Ранняя ветка vs ppkp2019 a02

- **Цель:** локализовать позднюю сервисную ветку.
- **Области/адреса:** ранние builders `0x89xx` против сервисного кластера `0x92EF/0x9489/0x94D6`.
- **Ожидание:** в 2019 больше транзакций через поздние сервисные подпрограммы.
- **Критерий:** хук/трасса вызовов показывает другой routing при схожем внешнем поведении.

## Ограничения

- Без эмулятора/железа невозможно окончательно доказать on-wire формат.
- Стендовые сценарии предполагают наличие точки наблюдения XDATA или трассировщика памяти.
