# Object record schema (рабочая версия)

Дата: 2026-04-24 (UTC).  
База записи: `XDATA 0x343D + idx*16`. Размер записи: **16 байт**.

## 1) Что подтверждено по коду

- В A04 присутствует явный helper вычисления адреса записи по индексу (`idx*16 + 0x343D`) по адресу `0x6759` и сходный helper чтения блока по `0x67AD` — **подтверждено**.
- Индекс объекта берется из `0x3298` и участвует в адресации записи — **подтверждено**.
- В A04 есть прямые обращения к смещениям `+0`, `+1`, `+2`, `+3`, `+5`, `+6` (`0x343D..0x3443`) — **подтверждено**.

## 3) Межпоколенная стабильность полей

### Стабильно межпоколенно (на текущем уровне)

- Факт существования записи 16 байт и индексной адресации `base + idx*16` — **подтверждено** для A03/A04, **вероятно** переносится на 2012/2019.
- Семантика «первые байты несут ключевое состояние объекта» — **сильная гипотеза**.

### Нестабильно/не доказано межпоколенно

- Точная роль каждого смещения `+2..+15` в 2012/2019 — **неизвестно**.
- Тождественность layout в поздней сервисной ветке `ppkp2019 a02` — **гипотеза** (прямых ссылок на `0x343D` в верхней странице не найдено).

## 4) Ограничения и следующий шаг

- Без трассировки выполнения (эмулятор/стенд) нельзя строго привязать значения к физическим событиям.
- Следующий шаг: построить cross-reference по всем опкодам, формирующим смещения внутри `idx*16` и разделить readers/writers по каждому байту записи.
